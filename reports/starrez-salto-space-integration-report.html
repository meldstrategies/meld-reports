StarRez to Salto Space Integration

Tenant Onboarding and Digital Key Issuance via JUSTin Mobile
Prepared for: Meld Strategies
Date: 31 March 2026
Prepared by: Meld Strategies — Smart Building Consulting
Classification: Internal

Executive Summary

This report examines the integration pathway between StarRez, a residential and property management platform, and Salto Space, an on-premises access control system, to enable automated tenant onboarding with digital key delivery via the Salto JUSTin Mobile app. The integration leverages the StarRez REST API (v2.0) for tenant data extraction and event monitoring, and the Salto Space SHIP (Salto Host Interface Protocol) for programmatic user creation and credential assignment within the access control system. A middleware integration layer is recommended to bridge the two platforms, mapping StarRez tenant creation events to Salto Space user provisioning and JUSTin Mobile digital key issuance. This approach enables a fully automated workflow where new tenants receive building access credentials on their smartphones without manual intervention by facility management staff.

Background

Commercial and residential property operators increasingly seek to automate the tenant onboarding process, eliminating manual steps in provisioning building access. When a new tenant is added to a property management system, the expectation is that their building access credentials — including digital keys for smartphone-based entry — should be provisioned automatically.

This report investigates how StarRez, used as the tenant and residential management system, can be integrated with Salto Space, an electronic access control platform, to achieve this automated workflow. The specific use case focuses on:

  1. A new tenant is created in StarRez (Entry record with an associated Booking)
  2. The tenant's details are automatically provisioned as a user in Salto Space
  3. Access permissions are assigned based on the tenant's allocated space
  4. A digital key is issued to the tenant via the Salto JUSTin Mobile app

System Overview

StarRez — Residential Management Platform

StarRez is a global residential management platform used by higher education institutions, student housing providers, and commercial property operators. It manages the complete resident lifecycle including applications, room allocation, bookings, billing, and check-in/check-out processes (StarRez).

Salto ProAccess Space is an on-premises access control management platform by Salto Systems. It manages electronic locks, user credentials, access permissions, and audit trails across buildings and campuses (Salto Developer).

Salto JUSTin Mobile

Salto JUSTin Mobile is a smartphone application (iOS and Android) that replaces physical access cards with digital keys. Keys are delivered Over-The-Air (OTA) from the Salto Space management software to the tenant's registered and verified mobile device. The phone uses Bluetooth Low Energy (BLE) or NFC to communicate with Salto electronic locks (Salto JUSTin Mobile User Guide).

Event Detection — Data Subscriptions

StarRez supports a Data Subscription mechanism that allows external systems to subscribe to data change events. When a monitored record changes (e.g., a new Entry is created, a Booking status changes), a message is generated that can be polled via the /getnextmessage/{subscriberID} endpoint (StarRez — Data Subscriptions). This provides the event-driven trigger needed to initiate the Salto Space provisioning workflow.

Additionally, StarRez has a certified Microsoft Power Automate connector (StarRez REST v1) that can be used to create scheduled or trigger-based flows — for example, polling for new bookings at defined intervals and triggering downstream API calls to Salto Space (StarRez Power Automate Announcement).

Salto Space Integration Interfaces

Salto Space provides several integration pathways, each suited to different use cases:

1. SHIP Protocol (Salto Host Interface Protocol)

SHIP is Salto's proprietary integration protocol, designed to allow third-party systems (BMS, CCTV, time and attendance, property management) to programmatically manage users, keys, and access permissions within Salto Space (Salto SHIP Documentation).

CapabilityDetail
Protocol TypeProprietary (Salto-defined message format)
TransportSTP (TCP/IP) or HTTP(S) — configurable
AuthenticationNone, Basic (HTTP), or Custom API Key
DirectionBidirectional — server and client modes
OperationsUser CRUD, key assignment/cancellation, access level management, door control, audit trail retrieval
LicensingLicense-dependent feature — requires SHIP licence from Salto
NDA RequiredYes — integration requires NDA with Salto and access to SHIP specification document
EncodingISO-8859-1 or UTF-8 (configurable)
Important: The SHIP protocol specification is not publicly available. Integrators must contact Salto Systems, sign a non-disclosure agreement (NDA), and obtain the SHIP specification document to develop against this interface. This is the recommended pathway for deep, automated integration between property management systems and Salto Space.

2. Seam API (Third-Party Cloud Bridge)

Seam provides a cloud-based API that acts as a bridge to on-premises Salto Space installations. It abstracts the SHIP protocol into a modern RESTful API with standardised endpoints for user management, access group assignment, and credential provisioning (Seam — Salto Space Integration).

Seam API ResourceSalto Space EquivalentDescription
Access System UserUser (Cardholder)Create and manage individuals with access permissions
Access GroupUser Access LevelCollections of users with shared permissions to specific entrances
EntranceAccess PointPhysical doors secured by Salto devices
CredentialKey (Physical or Digital)Key cards or mobile keys — max 1 card + 1 mobile key per user

The Seam integration requires a Seam Bridge device installed on the same network as the Salto Space server to relay API commands between the Seam cloud and the on-premises Salto Space installation.

3. Salto Space Hospitality API

Salto provides a dedicated Hospitality API for hotel and short-stay accommodation use cases. This API supports Apple Wallet key creation and management, guest check-in/check-out workflows, and key lifecycle operations (Salto Space Hospitality API). Authentication uses Bearer tokens issued by Salto.

Note: The Hospitality API currently supports only Apple Wallet keys for hotel guest rooms. Physical cards and staff keys are not supported through this API. For broader tenant management including JUSTin Mobile BLE keys, the SHIP protocol or Seam API bridge is required.

4. JUSTin Mobile SDK

For organisations building custom mobile applications, Salto provides the JUSTin Mobile SDK (iOS and Android). This SDK enables developers to embed digital key and BLE door-unlock capabilities directly into a branded tenant app (Axon.dev — JUSTin Mobile SDK Integration).

  • SDK is not publicly available — must be requested from Salto Systems
  • Typical integration timeline: 2–4 weeks for basic, longer for complex multi-user workflows
  • Requires iOS and/or Android developers plus a backend developer
  • All key data is encrypted end-to-end

Integration Architecture

Recommended Architecture

The integration between StarRez and Salto Space requires a middleware integration layer to orchestrate data flow between the two platforms. Three architectural options are available:

OptionApproachProsCons
Recommended
Option A: Direct SHIP Integration		 Custom middleware server communicates with StarRez REST API and Salto Space SHIP protocol directly Full control over operations; lowest latency; no third-party dependency; supports all SHIP operations including JUSTin key assignment Requires SHIP NDA and specification; more development effort; requires network access to Salto Space server
Alternative
Option B: Seam API Bridge		 Middleware communicates with StarRez REST API and Seam cloud API; Seam Bridge relays to Salto Space Modern RESTful API; well-documented; faster development; standardised credential management Requires Seam subscription and Seam Bridge hardware; adds cloud dependency; potential latency
No-Code
Option C: Power Automate + Seam		 Microsoft Power Automate uses StarRez connector for triggers, HTTP connector for Seam API calls No custom code required; visual workflow designer; leverages StarRez Power Automate connector Limited error handling; Power Automate Premium licence required; less flexible for complex logic

Step 1 — New Tenant Created in StarRez

Trigger Event

A property administrator creates a new Entry (tenant) record in StarRez via the web interface, self-service portal, or API. The Entry record includes the tenant's name, email, mobile phone number (in international format, e.g. +61412345678), and any relevant ID fields.

API Reference:

POST /StarRezREST/services/create/Entry { "NameFirst": "Jane", "NameLast": "Smith", "PortalEmail": "jane.smith@email.com", "ID1": "TNT-20260401-001" }

The system returns the new EntryID which serves as the unique identifier for all subsequent operations.

Step 2 — Booking Confirmed (Room Allocated)

Room Allocation

A Booking record is created linking the Entry to a specific Room Space, with defined check-in and check-out dates. This establishes the access period and physical location that will be mapped to Salto Space access permissions.

POST /StarRezREST/services/create/Booking { "EntryID": 12345, "RoomSpaceID": 501, "RoomLocationID": 42, "CheckInDate": "2026-04-15T00:00:00", "CheckOutDate": "2027-04-14T23:59:59", "EntryStatusEnum": "InRoom" }

Step 3 — Middleware Detects Event

Event Detection

The middleware layer detects the new booking via one of these methods:

  • Data Subscription Polling: The middleware polls /getnextmessage/{subscriberID} for new booking creation events
  • Scheduled Query: A periodic StarQL query (/query) retrieves recently created bookings with status "InRoom" or "Confirmed"
  • Power Automate Trigger: A scheduled flow polls the Get Booking action and filters for new records

Upon detection, the middleware retrieves the full tenant profile:

// 1. Get Entry (tenant details) GET /StarRezREST/services/select/Entry/12345 // 2. Get Entry Detail (mobile phone) GET /StarRezREST/services/select/EntryDetail?EntryID=12345 // 3. Get Booking (room and dates) GET /StarRezREST/services/select/Booking?EntryID=12345&_orderby=CheckInDate+desc&_top=1 // 4. Get Room Location (building/floor mapping) GET /StarRezREST/services/select/RoomLocation/42

Step 4 — User Created in Salto Space

Salto Space User Provisioning (via SHIP or Seam API)

The middleware creates a new user in Salto Space, mapping StarRez fields to Salto user attributes:

StarRez FieldSalto Space FieldNotes
NameFirstFirst NameDirect mapping
NameLastLast NameDirect mapping
EntryID or ID1External ID / GPF1Used for cross-reference and duplicate prevention
PortalEmailEmailUsed for notifications
Mobile Phone (from EntryDetail)Mobile Phone DataMust be in international format (+61...) for JUSTin key delivery
Booking.CheckInDateUser Activation DateWhen the user's key becomes active
Booking.CheckOutDateUser Expiration DateWhen the user's key expires

Via Seam API (Option B):

POST https://connect.getseam.com/acs/users/create { "acs_system_id": "acs_system_xxx", "full_name": "Jane Smith", "email_address": "jane.smith@email.com", "phone_number": "+61412345678", "access_schedule": { "starts_at": "2026-04-15T00:00:00+10:00", "ends_at": "2027-04-14T23:59:59+10:00" }, "external_type": "pms_user", "external_type_display_name": "Tenant" }
Mobile Phone Format: Salto requires the mobile number in international format with country code prefix and no spaces (e.g. +61412345678 for Australian numbers). The middleware must validate and reformat phone numbers from StarRez before passing to Salto Space (StayNTouch — Salto Mobile Integration).

Step 5 — Access Level Assigned

Access Group / Level Mapping

The middleware maps the StarRez Room Location to the corresponding Salto Space User Access Level. This requires a pre-configured mapping table maintained as part of the integration configuration:

StarRez Room LocationSalto Space User Access LevelAccess Points Included
Building A — Level 12"Level 12 Tenant"Main entrance, Lift lobby, Level 12 corridor, Allocated unit
Building A — Level 14"Level 14 Tenant"Main entrance, Lift lobby, Level 14 corridor, Allocated unit
Building B — Ground"Building B Ground Tenant"Building B entrance, Ground corridor, Allocated unit

Common area access (gym, mail room, bicycle storage) can be added as a separate base-level access group assigned to all tenants.

Via Seam API:

POST https://connect.getseam.com/acs/users/add_to_access_group { "acs_user_id": "acs_user_xxx", "acs_access_group_id": "acs_access_group_level12" }

Step 6 — JUSTin Mobile Digital Key Issued

Digital Key Delivery

Once the user is created in Salto Space with the correct mobile phone number and access levels, a digital key is assigned. This triggers the JUSTin Mobile key delivery process:

  1. Key Assignment: The middleware instructs Salto Space to assign a JUSTin Mobile key to the user (via SHIP command or Seam credential creation)
  2. OTA Delivery: Salto's cloud infrastructure delivers the encrypted digital key to the JUSTin Mobile app on the tenant's smartphone
  3. SMS Verification: Salto sends a verification code via SMS to the tenant's registered mobile number
  4. App Activation: The tenant downloads the JUSTin Mobile app (if not already installed), enters the verification code, and the key appears in the app
  5. Building Access: The tenant can now unlock doors by opening the JUSTin Mobile app, selecting the key, and presenting the phone to the Salto lock via BLE or NFC

Via Seam API (Mobile Key Credential):

POST https://connect.getseam.com/acs/credentials/create { "acs_user_id": "acs_user_xxx", "access_method": "mobile_key", "is_multi_phone_sync_credential": false }

Tenant-Side Experience:

  1. Tenant receives an SMS or email notification to download the JUSTin Mobile app
  2. Tenant opens the app, enters their mobile number
  3. Salto sends an SMS verification code
  4. Tenant enters the code — the app activates and displays available keys
  5. Tenant taps the green unlock button and presents phone to the lock

Recommendations

  1. Engage Salto Systems early — Initiate the NDA process and request the SHIP specification document. This is a prerequisite for the recommended direct integration (Option A) and typically takes 1–2 weeks.
  2. Evaluate Seam API as an accelerator — If rapid deployment is prioritised, the Seam API bridge (Option B) provides a well-documented REST API that abstracts the SHIP complexity. This is a good option if the integrator lacks prior SHIP experience.
  3. Validate mobile phone data quality — Ensure StarRez captures mobile numbers in international format (+61...) as this is critical for JUSTin Mobile key delivery. Implement format validation in the StarRez portal or middleware layer.
  4. Build the room-to-access-level mapping — Create and maintain a configuration table that maps StarRez Room Location IDs to Salto Space User Access Level names. This mapping should be version-controlled and auditable.
  5. Implement error handling and retry logic — The middleware should handle API failures gracefully, with retry logic, dead-letter queuing, and alerting for failed provisioning attempts.
  6. Test with pilot group — Deploy the integration with a small tenant cohort before full rollout to validate the end-to-end workflow, particularly the JUSTin Mobile key delivery and SMS verification flow.
  7. Plan for lifecycle events — Ensure the integration handles not just initial provisioning but also room changes, lease extensions, move-outs, and emergency access revocation.
  8. Document the integration — Maintain a living integration specification document covering field mappings, API credentials, error codes, and operational procedures for troubleshooting.
